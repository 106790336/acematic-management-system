# ACEMATIC运营管理系统 产品需求文档（PRD）

**版本：v2.1 子任务功能增强版**  
**日期：2026-04-12**  
**更新说明：在v2.0权限体系基础上，新增子任务分解功能**

---

## 一、系统概述

### 1.1 定位
为ACEMATIC（年营收4000万，12人团队）打造的闭环运营管理系统，实现"战略→计划→任务→子任务→执行"全链路可控、可追踪、可复盘。

### 1.2 核心设计原则
1. **权限精准控制** - 任务只有分配到的人才能操作
2. **任务可分解** - 执行者可将复杂任务分解为子任务
3. **来源唯一追溯** - 每个任务/子任务必须有明确来源
4. **战略对齐闭环** - 战略→计划→任务→子任务→执行完整关联
5. **零复制执行** - AI全权执行，系统自动化管理

## 二、权限控制体系

### 2.1 权限模型

#### 2.1.1 角色定义

| 角色 | 职责范围 | 可查看数据 |
|------|----------|-----------|
| **总经理** | 全公司战略与运营 | 所有数据 |
| **运营总监** | 运营中心管理 | 运营中心+下属部门 |
| **营销总监** | 营销中心管理 | 营销中心+下属部门 |
| **产品总监** | 产品中心管理 | 产品中心+下属部门 |
| **中心负责人** | 本中心人员与任务 | 本中心数据 |
| **部门经理** | 本部门人员与任务 | 本部门数据 |
| **普通员工** | 个人任务执行 | 仅限本人相关任务 |

#### 2.1.2 任务权限矩阵

```
权限维度：
┌─────────────────────────────────────────────────────────────┐
│                    任务操作权限                               │
├─────────────────────────────────────────────────────────────┤
│  查看权限    │ 谁可以查看任务详情                             │
│  编辑权限    │ 谁可以修改任务内容                             │
│  进度更新    │ 谁可以更新任务进度                             │
│  进度描述    │ 谁可以添加进度描述/备注                        │
│  审批权限    │ 谁可以审批任务申请/验收                        │
│  删除权限    │ 谁可以删除任务                                 │
│  分配权限    │ 谁可以给他人分配任务                           │
│  分解权限    │ 谁可以分解子任务【新增】                       │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 任务权限规则详解

#### 2.2.1 任务创建者（分配者）

| 操作 | 权限 | 说明 |
|------|------|------|
| 查看任务 | ✅ | 可查看自己创建/分配的所有任务 |
| 编辑任务 | ✅ | 可修改任务标题、截止时间、优先级（执行前） |
| 进度更新 | ❌ | 不可代为更新（需执行者本人操作） |
| 进度描述 | ✅ | 可添加指导/反馈描述 |
| 审批任务 | ✅ | 审批主动申请的任务 |
| 验收任务 | ✅ | 验收已完成的任务 |
| 删除任务 | ✅ | 任务未开始前可删除 |
| 重新分配 | ✅ | 可转派给他人 |
| **分解子任务** | ❌ | **不可代为分解**（需执行者本人）【新增】 |

#### 2.2.2 任务执行者（被指派人）【核心权限】

| 操作 | 权限 | 说明 |
|------|------|------|
| 查看任务 | ✅ | 仅可查看分配给自己的任务 |
| 编辑任务 | ❌ | 不可修改任务定义（可备注说明） |
| 进度更新 | ✅ | **唯一可更新进度的人** |
| 进度描述 | ✅ | **唯一可添加执行描述的人** |
| 提交完成 | ✅ | 提交完成待验收 |
| 申请延期 | ✅ | 可申请延期并说明原因 |
| **分解子任务** | ✅ | **【核心新增】执行者可分解子任务** |
| **管理子任务** | ✅ | **【核心新增】可分配/管理子任务的执行者** |

#### 2.2.3 上级领导（审批者）

| 操作 | 权限 | 说明 |
|------|------|------|
| 查看任务 | ✅ | 可查看下属所有任务 |
| 审批任务 | ✅ | 审批主动申请的任务 |
| 验收任务 | ✅ | 验收下属提交完成的任务 |
| 进度反馈 | ✅ | 可点评任务进度/质量 |
| 查看子任务 | ✅ | 可查看任务下的所有子任务 |
| 分解子任务 | ❌ | 不可代为分解 |

#### 2.2.4 同级/无关人员

| 操作 | 权限 | 说明 |
|------|------|------|
| 查看任务 | ❌ | 不可查看他人任务 |
| 任何操作 | ❌ | 无任何操作权限 |

### 2.3 子任务权限规则详解【新增章节】

#### 2.3.1 子任务权限设计原则

```
┌─────────────────────────────────────────────────────────────┐
│              子任务权限继承与独立原则                         │
├─────────────────────────────────────────────────────────────┤
│  1. 子任务作为独立任务实体，拥有完整的任务权限体系            │
│  2. 父任务执行者自动获得子任务分解权限                        │
│  3. 子任务执行者由父任务执行者指定，可不同于父任务执行者      │
│  4. 子任务执行者仅对自己负责的子任务有操作权限                │
│  5. 父任务执行者可查看所有子任务，但不一定能操作              │
│  6. 上级领导可查看父任务及其所有子任务                        │
└─────────────────────────────────────────────────────────────┘
```

#### 2.3.2 子任务权限矩阵【详细】

| 操作 | 父任务执行者 | 子任务执行者 | 上级领导 | 创建者 |
|------|-------------|-------------|---------|--------|
| 查看父任务 | ✅ | ✅(自己的) | ✅ | ✅ |
| 查看子任务 | ✅(全部) | ✅(自己的) | ✅(全部) | ✅ |
| 分解子任务 | ✅ | ❌ | ❌ | ❌ |
| 分配子任务执行人 | ✅ | ❌ | ❌ | ❌ |
| 更新子任务进度 | ❌ | ✅(自己的) | ❌ | ❌ |
| 更新父任务进度 | ✅ | ❌ | ❌ | ❌ |
| 删除子任务 | ✅(未完成) | ❌ | ❌ | ❌ |
| 验收子任务 | ❌ | ❌ | ✅ | ✅ |
| 关闭父任务 | ✅(子任务全完成) | ❌ | ✅ | ✅ |

#### 2.3.3 子任务与父任务的权限关系

```
权限继承链：
┌────────────────────────────────────────────────────────────┐
│  父任务执行者                                               │
│     ├── 有权分解子任务                                      │
│     │   └── 为每个子任务指定独立执行者                      │
│     │       └── 子任务执行者仅对该子任务有操作权限          │
│     ├── 可查看所有子任务进度                                │
│     ├── 可删除/修改子任务（未开始前）                       │
│     └── 父任务进度 = 子任务进度的加权平均                   │
└────────────────────────────────────────────────────────────┘
```

## 三、子任务功能详细设计【新增核心章节】

### 3.1 子任务核心概念

#### 3.1.1 什么是子任务

```
子任务定义：
┌─────────────────────────────────────────────────────────────┐
│  子任务是父任务的细化分解，具备以下特征：                   │
│                                                             │
│  1. 独立性：子任务是独立的任务实体，拥有完整任务属性        │
│  2. 关联性：子任务必须归属一个父任务                        │
│  3. 层级性：支持多级嵌套（子任务可再分解孙任务）            │
│  4. 权限独立：子任务执行者可不同于父任务执行者              │
│  5. 进度汇总：父任务进度由子任务进度自动计算                │
│  6. 完成依赖：父任务完成必须等待所有子任务完成              │
└─────────────────────────────────────────────────────────────┘
```

#### 3.1.2 何时使用子任务

| 场景 | 说明 | 示例 |
|------|------|------|
| **复杂任务分解** | 一个任务包含多个独立环节 | "完成XX酒店方案" → 分解为"需求调研"、"方案设计"、"成本核算"、"方案评审" |
| **多人协作** | 需要不同人员负责不同部分 | "新产品上线" → 分解为"技术开发"(技术部)、"文案准备"(市场部)、"渠道对接"(运营部) |
| **阶段性任务** | 有明确阶段的长期任务 | "年度审计" → 分解为"资料准备"、"现场审计"、"整改完成"、"报告出具" |
| **质量把控** | 需要分步验收的任务 | "软件交付" → 分解为"单元测试"、"集成测试"、"用户验收测试" |

### 3.2 子任务权限与操作规则

#### 3.2.1 谁可以创建子任务

```
创建权限：
┌─────────────────────────────────────────────────────────────┐
│  ✅ 父任务的执行者（被分配人）                              │
│     - 任务状态为"进行中"时即可分解                          │
│     - 无需上级审批即可分解                                  │
│     - 但上级可查看所有子任务                                │
│                                                             │
│  ❌ 任务创建者/分配者                                      │
│     - 不能直接代为分解                                      │
│     - 但可指导/建议执行者如何分解                           │
│                                                             │
│  ❌ 上级领导                                               │
│     - 不能直接代为分解                                      │
│     - 但可查看和点评                                        │
└─────────────────────────────────────────────────────────────┘
```

#### 3.2.2 子任务执行者指定规则

| 规则 | 说明 | 示例 |
|------|------|------|
| **自执行** | 父任务执行者可将自己设为子任务执行者 | "方案设计"由自己执行 |
| **转分配** | 父任务执行者可将子任务分配给其他人 | "技术开发"分配给技术同事 |
| **跨部门** | 子任务可分配给其他部门人员 | 市场部任务中的"技术开发"子任务分配给技术部 |
| **多层级** | 子任务执行者还可继续分解孙任务 | 技术同事将"技术开发"再分解为"前端"、"后端" |

#### 3.2.3 子任务状态与操作权限

| 子任务状态 | 可执行操作 | 操作人 | 状态流转 |
|-----------|-----------|--------|---------|
| **待分配** | 编辑、删除、指定执行人 | 父任务执行者 | → 待确认 |
| **待确认** | 确认接收、拒绝 | 子任务执行者 | → 进行中/已拒绝 |
| **进行中** | 更新进度、添加描述、分解孙任务、申请延期、提交完成 | 子任务执行者 | → 待验收/已暂停 |
| **已暂停** | 恢复执行 | 父任务执行者 | → 进行中 |
| **待验收** | 验收通过、退回修改 | 父任务执行者 | → 已完成/需修改 |
| **已完成** | 归档 | 父任务执行者 | - |
| **已拒绝** | 重新分配、删除 | 父任务执行者 | → 待分配/已删除 |
| **需修改** | 修改后重新提交 | 子任务执行者 | → 待验收 |

### 3.3 子任务与父任务的进度关联

#### 3.3.1 进度计算规则

```
父任务进度 = Σ(子任务进度 × 子任务权重) ÷ Σ(子任务权重)

示例：
父任务: "完成XX酒店方案" (总权重: 100%)
├─ 子任务1: 需求调研 (权重30%, 进度100%) → 贡献: 30%
├─ 子任务2: 方案设计 (权重40%, 进度75%) → 贡献: 30%
├─ 子任务3: 成本核算 (权重20%, 进度50%) → 贡献: 10%
└─ 子任务4: 方案评审 (权重10%, 进度0%) → 贡献: 0%

父任务总进度 = (30 + 30 + 10 + 0) / 100 = 70%
```

#### 3.3.2 权重设置规则

| 权重方式 | 说明 | 适用场景 |
|---------|------|---------|
| **手动权重** | 父任务执行者为每个子任务设置权重 | 各子任务工作量差异大 |
| **平均权重** | 系统自动平均分配权重 | 各子任务工作量相近 |
| **时间权重** | 根据预估工时自动计算权重 | 有明确的工时预估 |

#### 3.3.3 子任务完成与父任务验收

```
完成依赖链：
┌─────────────────────────────────────────────────────────────┐
│  子任务1完成 ──┐                                           │
│  子任务2完成 ──┼──► 所有子任务完成 ──► 父任务可提交完成   │
│  子任务3完成 ──┘                                           │
│                                                             │
│  【规则】                                                   │
│  • 只有所有子任务都达到"已完成"状态，父任务才能提交完成   │
│  • 父任务执行者提交完成后，进入原任务的验收流程           │
│  • 原任务创建者验收通过后，整个任务闭环完成               │
│  • 任一子任务被退回修改，父任务状态自动变为"进行中"     │
└─────────────────────────────────────────────────────────────┘
```

## 四、任务来源与战略对齐机制

### 4.1 任务来源体系（新增任务分解来源）

所有工作任务必须有明确来源，系统禁止创建无来源任务：

```
┌─────────────────────────────────────────────────────────────┐
│                    任务来源类型                              │
├─────────────────────────────────────────────────────────────┤
│  来源1: 上级分配 (assigned)                                  │
│    ├── 总经理分配给运营总监                                  │
│    ├── 运营总监分配给中心负责人                              │
│    └── 中心负责人分配给部门成员                              │

├─────────────────────────────────────────────────────────────┤
│  来源2: 主动提交 + 上级确认 (self_initiated)                 │
│    ├── 员工提交任务申请                                      │
│    └── 上级审核确认后生效                                    │
│    【权限】执行者申请，上级审批通过后方可操作                │
├─────────────────────────────────────────────────────────────┤
│  来源3: 战略计划分解 (plan_decomposition)                    │

│    ├── 子任务 → 孙任务（支持多级）                           │
│    └── 执行者分解，无需额外审批                              │
│    【权限】父任务执行者创建子任务，指定子任务执行者          │

## 六、API权限控制

### 6.1 权限校验中间件

所有任务相关API必须校验：
1. 用户是否登录
2. 用户是否有该任务的查看权限
3. 对于写操作：用户是否有该任务的写权限
4. 记录权限审计日志

### 6.2 关键API权限

| API | 方法 | 权限要求 | 说明 |
|-----|------|---------|------|
| /api/tasks | GET | 查看权限 | 获取任务列表 |
| /api/tasks | POST | 分配权限 | 创建新任务 |
| /api/tasks/:id/subtasks | POST | **执行者权限** | **分解子任务** |
| /api/tasks/:id/subtasks/:sid/progress | POST | **子任务执行者** | **更新子任务进度** |
| /api/tasks/:id/subtasks/:sid/complete | POST | **子任务执行者** | **提交子任务完成** |
| /api/tasks/:id/subtasks/:sid/approve | POST | **父任务执行者** | **验收子任务** |

## 七、验收标准

### 7.1 权限控制
- [ ] 任务创建后，非执行者无法更新进度
- [ ] 同级人员无法查看他人任务
- [ ] 任务未确认前，执行者无法操作

### 7.2 子任务功能【新增】
- [ ] 执行者可以将任务分解为多个子任务
- [ ] 执行者可以为每个子任务指定独立执行者
- [ ] 父任务进度根据子任务进度自动计算
- [ ] 所有子任务完成后，父任务才能提交完成
- [ ] 非执行者无法分解子任务

### 7.3 安全验收
- [ ] 权限审计日志完整记录
- [ ] 越权操作被阻止并记录
- [ ] 敏感信息按权限级别展示
